# TEAM_RULES

Правила командної роботи над проєктом (HTML + CSS, Vite, GitHub).

---

## 1. Git / GitHub Flow

### 1.1. Головна гілка

- Головна гілка: **`main`**.
- У `main` лежить **тільки стабільний і перевірений код**.
- Прямі коміти в `main` **заборонені** – тільки через Pull Request (PR).

### 1.2. Робочі гілки

- Для кожної задачі/секції створюється окрема гілка **від `main`**.
- Формат назви гілки:

  ```bash
  feature/header
  feature/about
  feature/footer
  Одна секція = одна гілка (включаючи адаптив).
  ```

Створення гілки:

bash Копіювати код git switch -c feature/header 1.3. Оновлення гілки перед пушем
Перед тим як пушити або створювати PR:

bash Копіювати код git checkout main git pull origin main git checkout
feature/header git merge main Якщо є конфлікти – вирішуємо локально, комітимо,
пушимо.

1.4. Pull Request Назва PR:

text Копіювати код feature: секція about (mobile, tablet, desktop) В описі PR
вказати:

що зроблено;

стан адаптиву (що покрито: mobile/tablet/desktop);

що залишилося доробити (якщо є);

бажано додати скріншоти (мінімум mobile + desktop).

PR мерджиться тільки після рев’ю від колеги.

Не мерджимо самі себе без погодження (як мінімум 1 approve).

2. Структура проєкту Базова структура:

text Копіювати код src/ partials/ header.html about.html footer.html css/
base.css header.css about.css footer.css img/ main.js main.css 2.1. HTML
Розмітка секцій – у src/partials/:

header.html, about.html, footer.html тощо.

Головний шаблон (якщо є) підключає partial-файли.

2.2. CSS Стилі секцій – у src/css/:

header.css, about.css, footer.css – один файл на секцію.

Загальні стилі – тільки в base.css:

reset / normalize;

шрифти;

кольори;

змінні (:root { --color-primary: ... });

спільні утиліти (.container, загальні кнопки тощо).

Повторювані стилі не дублюємо в секційних файлах – виносимо в base.css.

3. HTML-правила 3.1. Семантика Використовуємо семантичні теги:

header, main, section, footer, article, nav, ul, ol, li тощо.

Один <h1> на сторінку; у секціях – <h2>, далі за ієрархією.

3.2. Класи та іменування (BEM-підхід) Для кожної секції – свій блок:

html Копіювати код

<section class="about">
  <div class="container">
    <h2 class="about__title">Про нас</h2>
    <p class="about__text">...</p>
  </div>
</section>
Префікс класу завжди починається з назви секції:

text Копіювати код about**title about**text header**nav header**logo
footer\_\_link Модифікатори:

text Копіювати код about**title--accent header**btn--primary Клас .container:

використовується тільки для вирівнювання та обмеження ширини;

один спільний стиль у base.css.

3.3. Атрибути Порядок атрибутів (рекомендований):

html Копіювати код
<img class="about__image" id="team-photo" src="..." alt="Опис зображення" title="Назва" />
Використовуємо тільки подвійні лапки " ".

3.4. Доступність (Accessibility) У всіх img має бути осмислений alt, окрім чисто
декоративних зображень (alt="").

Не видаляємо outline у фокусних елементів (:focus) без заміни на власний
фокус-стиль.

Текст у кнопках та лінках має бути зрозумілим («Докладніше», «Замовити» тощо, а
не «Тут»).

4. CSS-правила 4.1. Іменування класів Починаємо з назви секції: header**,
   about**, footer\_\_ тощо.

Не використовуємо “голі” класи типу .title, .btn, .text в секціях – тільки з
префіксом блоків.

4.2. Структура та mobile-first Пишемо стилі в mobile-first підході:

css Копіювати код /_ мобілка _/ .about { /_ базові стилі _/ }

@media (min-width: 768px) { /_ планшет _/ .about { /_ адаптація _/ } }

@media (min-width: 1440px) { /_ десктоп _/ .about { /_ десктоп-адаптація _/ } }
Точки перелому (за замовчуванням):

min-width: 320px – мобільна базова;

min-width: 768px – планшет;

min-width: 1440px – десктоп.

4.3. Технічні обмеження Сітка – тільки flex (grid у цьому проєкті не
використовуємо).

Одиниці вимірювання: px, %.

Форматування коду – через .prettierrc (командами форматування в редакторі, не
руками).

Селектори для стилізації – тільки класи (без прив’язки до тегів і id).

Максимальна вкладеність селекторів: 2 рівні:

css Копіювати код .about\_\_list { ... }

.about\_\_list-item { ... }

/_ допуск: _/ .about**list-item .about**icon { ... } Не використовуємо
!important, окрім крайніх випадків:

якщо все-таки потрібно – додаємо коментар із поясненням:

css Копіювати код .about\_\_title { margin-bottom: 24px !important; /_
тимчасово: перекриваємо сторонні стилі _/ } 4.4. base.css та змінні У base.css
зберігаємо:

reset / normalize;

базові стилі body, html;

змінні кольорів і шрифтів:

css Копіювати код :root { --color-primary: #...; --color-bg: #...; --font-main:
'Roboto', sans-serif; }

body { font-family: var(--font-main); color: var(--color-primary); } У секційних
CSS не прописуємо “випадкові” кольори, беремо тільки зі змінних у base.css.

5. Зображення та ресурси Використовуємо оптимізовані формати:

фото: .webp, .jpg;

іконки, логотипи: .svg.

Не заливаємо в репозиторій оригінали по 5–20 МБ.

Перевіряємо розміри зображень: не використовуємо 4000px ширину, якщо на макеті
300px.

Шляхи до зображень мають бути відносними до папки src.

6. Коміти 6.1. Формат повідомлень Формат:

text Копіювати код feat(header): додано HTML і стилі мобільної версії
style(header): адаптив для планшета fix(header): виправлені відступи на мобілці
refactor(header): спрощено структуру розмітки docs: оновлено TEAM_RULES.md Коміт
має бути коротким і змістовним.

Одна логічна зміна – один коміт.

6.2. Мова комітів Одна мова для всієї команди: або українська, або англійська.

Домовляємося в команді один раз і далі дотримуємося.

7. Робота з репозиторієм (для новачків) 7.1. Як завантажити проєкт Встановити
   Node.js (версія LTS).

Клонувати репозиторій:

bash Копіювати код git clone
https://github.com/dmytro-dobrotvorskyi/project-08.git Перейти в папку проєкту:

bash Копіювати код cd project-08 Встановити залежності:

bash Копіювати код npm install 7.2. Запуск проєкту локально bash Копіювати код
npm run dev Сайт буде доступний за адресою, яку покаже консоль (зазвичай
http://localhost:5173/).

7.3. Збірка проєкту bash Копіювати код npm run build Це створить оптимізовану
продакшн-версію.

8. Робота з чужими змінами 8.1. Оновлення main Якщо хтось оновив main,
   підтягнути зміни:

bash Копіювати код git checkout main git pull origin main 8.2. Оновлення своєї
гілки bash Копіювати код git checkout feature/header git merge main Якщо виникли
конфлікти:

вирішуємо їх у редакторі;

перевіряємо роботу проєкту;

робимо коміт:

bash Копіювати код git add . git commit -m "fix(header): resolve merge
conflicts" git push origin feature/header 9. Що не пушимо в репозиторій Папки
node_modules/, dist/ – мають бути в .gitignore.

Локальні файли IDE/редактора (якщо не домовлено навпаки).

Тимчасові файли, особисті чернетки, тестові картинки.

10. Комунікація в команді Якщо щось ламається – пишемо в командний чат, не
    пушимо в main «якось працює».

Якщо сумніваєтесь у назві класу або рішенні – пишемо в чат, запитуємо.

Не змінюємо чужу секцію без попередження:

або пишемо автору в чат;

або описуємо зміни в PR з позначкою автора.

11. Checklists ✅ 11.1. Перед пушем HTML/CSS без помилок (VS Code, валідатор,
    W3C за потреби).

Адаптивність перевірена: мобілка / планшет / десктоп.

Оптимізовані зображення, немає зайвих картинок.

Видалений закоментований код і тимчасові класи.

Немає помилок у консолі браузера.

11.2. Перед створенням PR Локальна гілка оновлена від main (git pull origin
main + git merge main).

Всі зміни закомічені.

Коротка й зрозуміла назва PR.

У описі PR:

що зроблено;

стан адаптиву;

чи є відомі баги;

додані скріншоти (мінімум 2–3 основні стани).

Проєкт збирається і запускається без помилок.
